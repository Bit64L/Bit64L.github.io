动态连通性问题(Dynamic Connectivity)

当程序从输入中读取了整数对p,q时，如果p,q没有相连接，则连接p,q,否则跳过继续处理下一个整数对。图1用一个例子说明了这个过程。我们一般把这个问题叫做动态连通性问题。



图1

本篇博文采用网络方面的术语，将对象称为触点，将整数对称为连接，将等 价类称为连通分量或分量。

Union-find算法的API

方法 说明
void union(int p, int q) 在点p和点q之间添加一条连接
int find(int p) p所在的连通分量的标识符
boolean connected(int p, int q) 如果p和q存在于同一连通分量则返回true
int count() 连通分量的数量

API的实现

本文讨论三种实现，它们均是根据以触点为索引的id[]数组来确定两个触点是否在同一连通分量中的。

Quick-find算法

该方法是保证当id[p]等于id[q]时, p和q是连通的。

算法分析

Algorithm Initialize Union Find
QuickFind N N 1

Union()操作效率太低。 当使用quick-find算法来解决动态连通性问题并且最后只能到一个连通分量时，那么至少要调用N-1次union()操作，而一次union()操作需要最多访问2N+2次数组，则该算法的时间复杂度是O(n2)(quadratic)。

  • id[]数组的索引表示触点，值表示该触点的父亲
  • 根触点的父亲是它自己
  • 一个触点的根触点为id[id[id[...]]]

算法分析

  • 在最好情况下，find()操作只需要访问数组一次；最坏情况下，find()操作需要访问2N+1次数组。
  • 解决动态连通性问题时，我们可以构造一个最佳情况使得算法复杂度为线性级别；构造最差情况使得算法复杂度为平方级别。如图4.
  • quick-union可以看做是quick-find的一个改良，因为quick-find算法的union操作总是线性级别的;但是我们并不能保证在任何情况下quick-union都比quick-find快得多。

加权的quick-union算法

为了保证树的深度尽可能的小，我们可以在进行union()操作时，总是将小树归并的大树上去（树的大小由树所包含结点的数目来确定）。增加一个sz数组来记录树的大小。

算法分析

  • 对于加权quick-union算法和N个触点，在最坏情况下find()、connnected()和union()的成本的增长数量级为logN。
  • 加权quick-union算法能够在合理的时间范围内解决大规模动态连接性问题，其最坏情况下的复杂度为NlogN.
